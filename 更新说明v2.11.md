# v2.11 已调度 UE 的 SU 吞吐 CDF 与 paired-case 调试更新说明

## 1. 更新目标

本轮修改解决两个问题：

1. 增加最终已调度 UE 在所选服务波束上的单用户（SU）吞吐统计和 CDF，用于判断不同调度方案最终选择了怎样的单用户吞吐人群；
2. 增加相同物理调度结果的 paired-case 调试检查，用于定位 schedule 相同但 goodput 不同的问题究竟从 link 顺序、realized SINR、OLLA、MCS、TBLER、ACK 随机数还是 goodput 开始分叉。

本说明只记录代码功能修改，不记录具体实验 YAML 的参数数值调整。

## 2. 已调度 UE 的 SU 吞吐统计与 CDF

### 2.1 修改了什么

仿真现在会对每个 drop、每个 evaluation case 的最终已调度 UE，统计其实际所选 service beam 对应的：

- UE ID；
- beam index 和可读 beam ID；
- SU SNR；
- SU MCS；
- SU outage；
- SU throughput。

新增输出：

```text
metrics/scheduled_ue_su_throughput.csv
metrics/scheduled_ue_su_throughput_summary.csv
figures/scheduled_ue_su_throughput_cdf.png
```

其中：

- `scheduled_ue_su_throughput.csv` 是逐 drop/case/UE 的原始样本；
- `scheduled_ue_su_throughput_summary.csv` 输出各 case 的样本数、平均值、P05、P50、P95 和 SU outage 比例；
- `scheduled_ue_su_throughput_cdf.png` 对比各 evaluation case 的已调度 UE SU 吞吐 CDF。

相同汇总数据也会写入：

```text
metrics/summary.json
```

中的 `_scheduled_ue_su_throughput` 节点。

### 2.2 统计口径

SU throughput 使用最终被调度 UE 的实际所选 service beam，但不加入其它共调度 UE 的干扰：

```text
最终 schedule
  -> 找到 UE report 中所选 beam 的 ServiceCandidate
  -> 读取 standalone SU SNR / SU MCS / SU outage
  -> 使用当前 link adapter 和 PDSCH 资源配置换算 rate_mbps
  -> SU outage 样本记为 0 Mbps
```

该指标与下面两个指标不同：

1. 它不是 scheduler 输出的 `predicted_rate_mbps`。特别是 full-Gamma case 的 predicted rate 可能已经包含预测的多用户干扰；
2. 它不是 `link_tti.csv` 中的实际 goodput。实际 goodput 包含真实多用户干扰、EESM、OLLA、实际 MCS、TBLER 和 ACK/NACK。

因此，该 CDF 主要用于观察调度器的用户选择倾向，而不是替代实际吞吐评估。

### 2.3 怎么修改的

在 `beam_sls/sim.py` 中新增：

```python
build_scheduled_ue_su_throughput_rows(...)
summarize_scheduled_ue_su_throughput(...)
```

每个 case 完成调度后，代码立即根据最终 `sched.links` 和对应 UE report 生成 SU 吞吐样本。所有 drop 完成后统一写 CSV、写入 summary，并调用已有的通用 `plot_cdf()` 生成图片。

若最终 schedule 中出现 report 不包含的 `(UE, beam)`，代码会直接报错，避免静默使用错误的 SU 数据。

### 2.4 改完后怎么用

该功能默认随正常仿真运行，不需要增加配置开关。继续使用原有命令即可，例如：

```powershell
python -m beam_sls.run `
  --config <CONFIG.yaml> `
  --out <RUN_DIR>
```

运行完成后重点查看：

```text
<RUN_DIR>/figures/scheduled_ue_su_throughput_cdf.png
<RUN_DIR>/metrics/scheduled_ue_su_throughput_summary.csv
```

需要检查某个异常点时，再根据 `scheduled_ue_su_throughput.csv` 中的 `drop`、`case_id`、`ue_id` 和 `beam_index` 回溯。

## 3. 相同 schedule 的 paired-case 调试

### 3.1 修改了什么

`link_tti.csv` 新增两个诊断字段：

```text
ack_random_uniform
link_position
```

- `ack_random_uniform` 是本次 TBLER 抽样实际使用的均匀随机数；
- `link_position` 是该 UE 在 `schedule.links` 中的遍历位置。

ACK 判定仍然是：

```python
ack = int(ack_random_uniform > tbler)
```

本次修改只是先保存原本已经生成的随机数再执行同一个判定，不改变 ACK、OLLA、MCS 或 goodput 逻辑。

另外新增紧凑 paired-case 检查。仿真结束后，它会按指定 case 对比较：

1. 每个 drop 的 `(ue_id, beam_index)` 集合是否相同；
2. `schedule.links` 顺序是否相同；
3. `link_tti.csv` 的 `(drop, tti, ue_id)` 行是否一一对齐；
4. `beam_index`；
5. `link_position`；
6. `effective_sinr_db`；
7. `olla_offset_db`；
8. `mcs_selection_sinr_db`；
9. `actual_mcs`；
10. `tbler`；
11. `ack_random_uniform`；
12. `ack`；
13. `goodput_bits`。

新增输出：

```text
metrics/paired_case_debug.txt
```

同样的内容也会以 `[paired-debug]` 前缀打印到终端。

### 3.2 为什么需要记录 link 顺序

当前每个 case 会恢复同一个 NumPy ACK 随机数流，并通过 `(case_id, ue_id)` 隔离 OLLA 状态。

但是随机数仍按下面的循环顺序消费：

```text
for tti:
  for link in schedule.links:
    draw ACK random number
```

因此：

- schedule 集合和顺序都相同：同一个 UE 会获得同一个随机数；
- schedule 集合相同但顺序不同：随机数流相同，但同一个随机数可能分配给不同 UE。

新增的 `link_position` 和 `ack_random_uniform` 可以直接区分这两种情况。

### 3.3 怎么修改的

在 `beam_sls/link.py` 中：

- 将 `schedule.links` 循环改为带 `link_position` 的枚举；
- 将 `rng.uniform()` 的结果保存为 `ack_random_uniform`；
- 将两个字段写入每条正式统计 TTI 的 `LinkEvalRow`。

warmup TTI 仍然正常消耗随机数和更新 OLLA，但不写入 `link_tti.csv`。如果正式 `tti=0` 的 `olla_offset_db` 已经不同，说明分叉发生在隐藏的 warmup 阶段。

在 `beam_sls/sim.py` 中新增：

```python
build_paired_case_debug_lines(...)
```

同时保存每个 drop/case 的有序 `(UE, beam)` 列表。全部链路评估结束后，调试函数只检查配置指定的 case 对，并输出：

- schedule 集合/顺序是否一致；
- 两侧行数和无法对齐的 row key 数；
- 各字段不一致的行数；
- 浮点字段最大绝对差；
- 第一处分叉的 drop、TTI、UE、字段和两侧取值；
- 第一处分叉行的 link position 和 ACK 随机数。

### 3.4 改完后怎么用

在配置中加入：

```yaml
analysis:
  paired_case_debug:
    enabled: true
    pairs:
      - [baseline__greedy, threshold_conflict_set__greedy]
      - [topk_conflict_id__hard_conflict_greedy, threshold_conflict_set__hard_conflict_greedy]
```

case 名称必须与 `evaluation.matrix` 生成的 `case_id` 完全一致，格式通常为：

```text
<feedback_scheme>__<algorithm>
```

正常启动仿真即可。仿真结束后，终端会出现：

```text
[paired-debug] begin
...
[paired-debug] end
```

定位问题时，只需复制这一小段输出，不需要先上传完整的 `link_tti.csv`。

同一份内容可以从下面的文件获取：

```text
<RUN_DIR>/metrics/paired_case_debug.txt
```

### 3.5 输出解读

#### 情况 1：完全一致

```text
schedule_set_equal=1
schedule_order_equal=1
result=EXACT_MATCH
```

表示两组 case 的物理调度、链路顺序和逐 TTI 评估结果完全一致。此时 summary goodput 也应一致。

#### 情况 2：集合相同但顺序不同

```text
schedule_set_equal=1
schedule_order_equal=0
```

结合：

```text
link_position>0
ack_random_uniform>0
```

可确认共同随机数流因为 link 遍历顺序不同而分配给了不同 UE。

#### 情况 3：从 realized SINR 开始不同

```text
effective_sinr_db>0
```

说明物理调度并非真正相同，或相同干扰项采用了不同求和顺序并恰好产生可观察的浮点差异。

#### 情况 4：正式 TTI 一开始 OLLA 就不同

第一处分叉若出现在：

```text
tti=0
field=olla_offset_db
```

说明差异已经在 `olla_warmup_tti` 阶段产生。

#### 情况 5：到 TBLER 都相同，ACK 随机数不同

```text
tbler=0
ack_random_uniform>0
```

说明链路物理结果和链路适配相同，差异来自 ACK 随机数没有按相同 UE 对齐。

#### 情况 6：逐行完全一致但 summary 不同

若 paired debug 返回 `EXACT_MATCH`，但 `summary.json` 中 goodput 仍不同，则问题已经缩小到汇总逻辑或结果文件混用，需要进一步检查输出目录和 summary 生成过程。

### 3.6 关闭调试

问题定位完成后可设置：

```yaml
analysis:
  paired_case_debug:
    enabled: false
```

关闭后不再生成 `paired_case_debug.txt`，但 `link_tti.csv` 中保留两个轻量字段，便于结果复核。

## 4. 有效占用带宽统一由 PRB 和 SCS 推导

### 4.1 修改了什么

删除独立的 `system.bandwidth_mhz` 配置。有效占用带宽现在唯一计算为：

\[
B_{\mathrm{occupied}} =
\mathrm{pdsch.num\_prbs}\times12\times
\mathrm{system.subcarrier\_spacing\_khz}\times10^3.
\]

同一个结果同时用于：

- numpy fallback 信道的频率响应采样跨度；
- Sionna TR 38.901 CIR 到频域信道的频率跨度；
- Sionna channel model 的 sampling frequency；
- 热噪声积分带宽。

以 `132 PRB` 和 `120 kHz` 为例，信道和噪声现在都会使用 `190.08 MHz`，不会再出现
20 MHz 信道频率选择性与 190.08 MHz 噪声/TBS 混用。

### 4.2 怎么修改的

在 `beam_sls/utils.py` 中新增：

```python
occupied_bandwidth_hz(cfg)
```

该函数读取：

```yaml
system:
  subcarrier_spacing_khz: 120.0

pdsch:
  num_prbs: 132
```

并校验二者均大于零。`beam_sls/channel.py` 的两个信道 backend 和
`beam_sls/sim.py` 的噪声计算全部调用这一函数，不再分别计算或读取独立带宽字段。

解析后的带宽会写入：

```text
resolved_config.yaml:
  _resolved.occupied_bandwidth_hz
  _resolved.occupied_bandwidth_mhz

metrics/drops.csv:
  occupied_bandwidth_mhz
```

默认配置和 `configs/` 下的仿真 YAML 均已删除 `bandwidth_mhz`。

### 4.3 改完后怎么用

配置中只需要给出 SCS 和 PRB 数：

```yaml
system:
  subcarrier_spacing_khz: 120.0

pdsch:
  num_prbs: 132
```

不要再配置 `bandwidth_mhz`。正常运行后，可检查：

```text
resolved_config.yaml
metrics/drops.csv
```

确认派生的 `occupied_bandwidth_mhz` 是否符合预期。
启动日志也会打印 `occupied bandwidth=... MHz` 及其 PRB/SCS 派生来源。

## 5. 兼容性

- 不改变 scheduler 的候选选择、目标函数或最终 schedule；
- 不改变 realized SINR、EESM、ILLA、TBLER、ACK/NACK 和 goodput 算法；
- 不改变已有 CSV 字段，只在 `link_tti.csv` 末尾增加诊断列；
- SU 吞吐统计只增加输出，不参与 scheduler 决策；
- paired-case debug 默认关闭，不影响未配置该功能的仿真；
- `bandwidth_mhz` 不再参与计算，信道与噪声带宽统一由 PRB 和 SCS 决定；
- 原有绘图和 summary 字段保持兼容。
