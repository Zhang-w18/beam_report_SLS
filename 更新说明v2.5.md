## 1. 结论摘要

### 1.1 当前三站点配置的性质

`configs/v2_three_site_global_36ue.yaml` 当前仍然是一个偏功能验证性质的配置：

```text
3 sites × 3 sectors/TRPs/site × 4 TXRUs/TRP = 36 TX units
4 UE/sector × 9 sectors = 36 candidate UEs
service_beam_top_k1 = 56
num_drops = 1
num_tti_per_drop = 10 measured TTIs
olla_warmup_tti = 100 warmup TTIs
```

v2.5 已增加 100 个 OLLA warmup TTI，但没有改变该配置中的 36 UE 和 56 个候选服务 beam。因此，这个配置仍不应直接作为统计稳定的正式性能评估配置。

### 1.2 已完成的 v2.5 修改

v2.5 已实现：

- 每个 drop 可配置 OLLA warmup TTI 数；
- warmup 期间正常抽样 ACK、选择 MCS、更新 OLLA；
- warmup 数据不进入吞吐、BLER、CDF 和 CSV 统计；
- `num_tti_per_drop` 继续表示正式统计 TTI 数；
- 支持命令行覆盖 warmup 数；
- `metrics/drops.csv` 记录 warmup 和正式统计 TTI 数；
- 新增独立离线 CDF 脚本，可从已有 CSV 重新画图；
- 可选择方案、指标、字体大小、颜色、线型、线宽、marker、图例和输出格式；
- 支持在同一张图叠加 baseline 无干扰 SU 参考曲线和其他方案；
- 可选导出绘图实际使用的 ECDF 原始点；
- 文字分析脚本会记录 `drops × warmup × measured TTI`。

### 1.3 尚未修改的部分

v2.5 没有修改：

- 三站点配置的 `service_beam_top_k1: 56`；
- 三站点配置的 36 个候选 UE；
- 每个 drop 只调度一次；
- 同一个 drop 内调度集合和信道保持不变；
- 每 TTI 或周期性重新调度；

这些问题仍需在后续版本中根据仿真目标继续处理。

## 2. 为什么当前配置上报 56 个服务波束

### 2.1 当前码本规模

三站点全局配置中：

```text
9 TRPs
× 4 TXRUs/TRP
× 8 beams/TXRU
= 288 network-wide TX beams
```

一个 TRP 内有：

```text
4 TXRUs × 8 beams/TXRU = 32 beams/TRP
```

但配置使用：

```yaml
scheduler:
  domain_mode: global

feedback:
  service_beam_top_k1: 56
  oracle_service_beam_top_k: 56
```

`global` 模式下，当前代码没有把候选服务 beam 限制在 UE 的 serving TRP 或 serving site。UE 会从全网 288 个 beam 中按 SU SNR 排序，选择 Top-56。

配置注释已经说明，56 不是根据实际反馈 bit budget 推导出来的，而是通过固定随机种子的实际调度路径试出来的：需要扩大候选集，才能让候选 beam 覆盖全部 36 个 TX unit，并使 baseline 调度满 36 个 UE。

### 2.2 为什么这个设置不适合作为有限反馈方案

当前代码没有计算或限制反馈 bit 数。设置 `K1=56` 时，每个 UE 至少需要上报：

- 56 个候选服务 beam ID；
- 每个候选的 SU SNR/CQI/MCS 信息；
- `topk_conflict_id` 方案中，每个服务 beam 还带 `K2=3` 个冲突 beam ID，最多对应 168 个冲突 ID；
- `threshold_conflict_set` 的冲突集合长度还可能更大。

因此，当前的 56 更接近“为了扩大调度搜索空间而使用的大候选集合”，而不是严格受限的 UE 反馈设计。

### 2.3 建议的上报规模

没有独立反馈格式和 bit budget 时，不存在唯一正确的 K1。建议使用以下方式做敏感性分析：

```text
K1 = 1, 2, 4, 8
```

可把 `K1=16` 作为较高反馈量参考，但不建议直接用 56 作为主要有限反馈结论。

如果仍允许跨 TRP 选择服务 beam，更合理的结构是：

```text
Top 1–2 candidate TRPs × Top 2–4 beams/TRP
```

这样总候选数仍可控制在 4–8。还应将“全网调度域”和“UE 允许上报的服务域”拆成两个独立配置，避免 `global` 调度自动等价于从全网 288 个 beam 上报候选。

## 3. 为什么候选 UE 不应该只有 36 个

### 3.1 36 是并发容量上限，不是撒点数量

当前配置：

```yaml
ue_drop:
  num_ut_per_sector: 4

scheduler:
  max_mu_order: 36
```

因此：

```text
4 UE/sector × 9 sectors = 36 candidate UEs
```

而物理调度容量也是 36。候选 UE 数恰好等于容量时，调度器缺少多用户选择空间。`max_mu_order: 36` 应理解为最多同时调度 36 个 UE，而不是只能撒 36 个 UE。

完全可以使用：

```yaml
ue_drop:
  num_ut_per_sector: 8   # 全网 72 UE
```

或：

```yaml
ue_drop:
  num_ut_per_sector: 10  # 全网 90 UE
```

最终调度结果仍最多为 36 个 UE。

### 3.2 greedy 是否会少于 36 个 UE

当前 greedy 的逻辑是：

1. 从空调度集合开始；
2. 尝试加入每个未使用 UE 的每个候选 beam；
3. 选择能够带来最大正效用增量的 `(UE, beam)`；
4. 如果不存在正增量候选则停止；
5. 否则继续，直到达到 `max_mu_order=36`。

因此代码形式上已经满足：

```text
如果增加用户能提高效用，则继续调度；
如果所有新增用户都不能提高效用，则少于 36 个也可以停止。
```

但不同反馈方案的效用模型不同：

- `full_gamma` 会在预测效用中考虑用户间干扰，增加 UE 可能降低已有 UE 的速率，因此可能提前停止；
- `baseline` 基本使用各候选的 SU rate 相加，未知干扰按零处理，只要还有空闲 TX unit 和正速率候选，通常会继续填满；
- proposed 方案依赖冲突惩罚，但固定 `lambda=0.35 Mbps` 相对数百 Mbps 的候选链路速率通常较小，也可能倾向于填满。

所以“允许少于 36”已经存在，但 baseline 的效用模型未必能可靠判断真实 MU 干扰下是否应该停止。

### 3.3 增加 UE 数后的复杂度

对固定 beam 数和频域采样点，信道生成及 Gamma 测量的内存和计算量大致随 UE 数线性增加。greedy 对所有 UE 候选进行多轮扫描，UE 越多、K1 越大，调度耗时也越长。

但从 `K1=56` 降到 `K1=4` 后，每 UE 候选扫描量减少约 14 倍。因此：

```text
90 UE × K1=4 = 360 candidate pairs
36 UE × K1=56 = 2016 candidate pairs
```

只看调度候选数量，增加 UE 同时显著降低 K1 未必更慢。完整 Gamma 测量仍会随 UE 数增加。

exhaustive 调度的组合复杂度会快速增长，不适合直接用于 72–90 UE、全局 36 阶调度；正式大规模运行应使用 greedy 类算法。

## 4. 当前蒙特卡洛仿真和 TTI 时序

### 4.1 每个 drop 做什么

当前主流程对每个 drop 执行：

1. 从 master RNG 产生该 drop 的独立随机种子；
2. 重新生成站点/sector 下的 UE 位置；
3. 重新生成该 drop 的信道；
4. 重新计算服务功率、SU SNR、MCS 和 Gamma；
5. 为每个反馈方案生成 UE report；
6. 每个方案只执行一次调度；
7. 使用这一个固定 schedule 运行 warmup TTI；
8. 继续使用同一个 schedule 运行正式统计 TTI；
9. 进入下一个 drop 时重新生成 UE 位置和信道，并重置 OLLA。

### 4.2 同一个 drop 内什么保持不变

当前同一 drop 的全部 warmup 和正式统计 TTI 中：

- UE 位置不变；
- 信道不变；
- 调度 UE 集合不变；
- 每个 UE 的服务 beam 不变；
- 真实逐频点 SINR 网格不变。

TTI 间变化的主要是：

- 随机 ACK/NACK；
- OLLA offset；
- 由 predicted SINR 减去 OLLA backoff 后重新选择的实际 MCS；
- ACK 成功时的实际 goodput。

`speed_kmh` 当前没有形成同一个 drop 内逐 TTI 的信道演化。`harq_enabled` 仍是预留字段，没有实现 HARQ 合并或重传。

### 4.3 对单用户长期吞吐的影响

每个 drop 只调度一次意味着，未选中的 UE 在这个 drop 的全部 TTI 都是零吞吐。`metrics/ue_goodput.csv` 会正确地保留这些零值，避免幸存者偏差。

但当候选 UE 从 36 增加到 72 或 90 后，如果仍只调度一次，大量 UE 会在整个 drop 内一直为零。这种 CDF 表示“一次静态调度选择后的吞吐”，不等价于真实系统经过很多 TTI 轮转调度后的长期用户吞吐。

如果目标是研究长期单用户吞吐、公平性或 PF，应进一步实现：

- 每 TTI 或每若干 TTI 重新调度；
- PF 平均吞吐在同一个 drop 内持续更新；
- 可选的信道或测量周期更新；
- warmup 后再开始统计长期用户吞吐。

这部分不在本次 v2.5 修改范围内。

## 5. v2.5 OLLA warmup 功能

### 5.1 配置字段

新增字段：

```yaml
link_abstraction:
  olla_enabled: true
  olla_step_db: 0.1
  olla_warmup_tti: 100
```

默认配置中：

```yaml
olla_warmup_tti: 0
```

用于保证旧配置结果和运行时间保持兼容。三站点全局配置中已经显式设置：

```yaml
link_abstraction:
  olla_warmup_tti: 100
```

负数配置会直接报错。

### 5.2 warmup 的统计语义

假设：

```yaml
system:
  num_tti_per_drop: 500

link_abstraction:
  olla_warmup_tti: 100
```

则每个方案、每个 drop 的链路循环实际执行：

```text
100 warmup TTIs + 500 measured TTIs = 600 TTIs
```

其中前 100 个 TTI：

- 抽样 ACK/NACK；
- 更新 OLLA offset；
- 重新选择 MCS；
- 不写入 `metrics/link_tti.csv`；
- 不进入系统平均吞吐；
- 不进入 UE 平均吞吐；
- 不进入 ACK rate、BLER、SINR 和吞吐 CDF。

后 500 个正式统计 TTI 写入 CSV，`tti` 编号仍是：

```text
0, 1, ..., 499
```

因此旧后处理脚本不需要把 warmup 行再次过滤掉。

### 5.3 OLLA 更新规则

当前 offset 是从 predicted SINR 中减去的 backoff：

```text
mcs_selection_sinr_db = predicted_sinr_db - olla_offset_db
```

目标 BLER 为 `target`、步长为 `step` 时：

```text
ACK:  offset -= step
NACK: offset += step × (1-target)/target
```

warmup 和正式统计阶段使用完全相同的更新逻辑。进入下一个独立 drop 时，OLLA offset 重新从 0 开始，不继承上一个随机 UE/channel realization 的状态。

### 5.4 配置文件运行方法

直接修改 YAML：

```yaml
system:
  num_drops: 50
  num_tti_per_drop: 500

link_abstraction:
  olla_enabled: true
  olla_step_db: 0.1
  olla_warmup_tti: 100
```

运行：

```bash
python -m beam_sls.run \
  --config configs/v2_three_site_global_36ue.yaml \
  --out runs/v2_5_three_site \
  --num-drops 50 \
  --num-tti 500
```

### 5.5 命令行覆盖 warmup

无需修改 YAML，也可以使用：

```bash
python -m beam_sls.run \
  --config configs/v2_three_site_global_36ue.yaml \
  --out runs/v2_5_three_site_warmup200 \
  --num-drops 50 \
  --num-tti 500 \
  --olla-warmup-tti 200
```

命令行参数会覆盖 YAML 中的 `link_abstraction.olla_warmup_tti`。

### 5.6 如何确认 warmup 生效

运行日志会显示：

```text
[run] drops=50, warmup_tti/drop=100, measured_tti/drop=500, ...
```

`metrics/drops.csv` 新增：

```text
olla_warmup_tti
num_measured_tti
```

`resolved_config.yaml` 会保留最终生效的 `olla_warmup_tti`。

注意：固定使用 100 个 warmup TTI 不等价于数学上证明 OLLA 已经收敛。建议分别尝试 0、50、100、200、500 等 warmup，检查正式统计阶段开始时的 OLLA offset、ACK rate 和平均 TBLER 是否已经对 warmup 长度不敏感。

## 6. baseline 无干扰 SU 吞吐参考

### 6.1 参考方案名称

scheme 名称为：

```text
baseline_no_interference_upper_bound
```

启用条件：

```yaml
feedback:
  schemes: [full_gamma, baseline, topk_conflict_id, threshold_conflict_set]

analysis:
  baseline_no_interference_upper_bound: true
```

必须包含 `baseline`，该参考方案才会生成。

### 6.2 它具体代表什么

实现过程是：

1. 先按正常 baseline 得到固定的 `(UE, beam)` 调度集合；
2. 不重新选 UE；
3. 不重新选 beam；
4. 重新运行链路评估；
5. SINR 分母只保留热噪声，强制忽略其他已调度 beam 的干扰；
6. 使用相同的 predicted MCS、ACK/BLER、TBS 和 OLLA 流程得到 goodput；
7. baseline 与该无干扰参考从同一个链路评估 RNG 状态开始，使用共同随机数降低抽样噪声。

因此它是：

```text
baseline 原调度集合的无波束间干扰 SU 诊断参考
```

它不是在无干扰条件下重新搜索所有 UE/beam 得到的全局最优调度。

### 6.3 原始数据

该 scheme 会写入：

```text
metrics/link_tti.csv
metrics/ue_goodput.csv
metrics/summary.csv
metrics/summary.json
```

主仿真生成的以下图中也会与其他方案一起出现：

```text
figures/link_goodput_cdf.png
figures/ue_goodput_cdf.png
figures/effective_sinr_cdf.png
```

## 7. v2.5 独立 CDF 绘图脚本

### 7.1 脚本用途

新增：

```text
scripts/plot_cdf.py
```

该脚本只读取已有 run 的 `metrics/*.csv`：

- 不重新生成 UE；
- 不重新生成信道；
- 不重新调度；
- 不重新抽样 ACK；
- 可以对同一份原始数据反复修改图形样式。

支持 PNG、PDF 和 SVG，输出格式由文件扩展名决定。

### 7.2 支持的 CDF 指标

```text
link_goodput
ue_goodput
effective_sinr
tbler
olla_offset
reported_su_snr
reported_max_su_snr
```

含义如下：

| metric | 数据文件 | 样本定义 |
|---|---|---|
| `link_goodput` | `link_tti.csv` | 每条已调度链路、每个正式统计 TTI 的 goodput |
| `ue_goodput` | `ue_goodput.csv` | 每个 `(drop, UE)` 跨正式统计 TTI 的平均 goodput，未调度 UE 为零 |
| `effective_sinr` | `link_tti.csv` | 每条已调度链路、每个正式统计 TTI 的 effective SINR |
| `tbler` | `link_tti.csv` | 链路 TBLER |
| `olla_offset` | `link_tti.csv` | 正式统计阶段 OLLA backoff |
| `reported_su_snr` | `su_snr_samples.csv` | 每个 UE 上报的每个候选 beam，一个候选一个样本 |
| `reported_max_su_snr` | `su_snr_max_per_ue.csv` | 每个 UE 的最大上报 SU SNR，一个 UE 一个样本 |

`link_goodput` 与 `ue_goodput` 不能混为一谈。研究单用户长期平均吞吐时，应优先使用 `ue_goodput`；研究每次链路传输的瞬时吞吐分布时，可使用 `link_goodput`。

### 7.3 快速画单用户吞吐 CDF

把 full-gamma、baseline 和 baseline 无干扰参考画在同一张图：

```bash
python scripts/plot_cdf.py runs/YOUR_RUN \
  --metric ue_goodput \
  --schemes \
    full_gamma \
    baseline \
    baseline_no_interference_upper_bound \
  --font-size 13 \
  --output runs/YOUR_RUN/figures/ue_goodput_compare.pdf
```

### 7.4 画全部反馈方案和无干扰参考

```bash
python scripts/plot_cdf.py runs/YOUR_RUN \
  --metric ue_goodput \
  --schemes \
    full_gamma \
    baseline \
    baseline_no_interference_upper_bound \
    topk_conflict_id \
    threshold_conflict_set \
  --output runs/YOUR_RUN/figures/ue_goodput_all.png
```

`--schemes` 的顺序也是画线和图例顺序。

### 7.5 查看某个指标中有哪些 scheme

```bash
python scripts/plot_cdf.py runs/YOUR_RUN \
  --metric ue_goodput \
  --list-schemes
```

如果旧 run 中没有 `baseline_no_interference_upper_bound`，脚本会报告当前 CSV 中实际存在的 scheme。

### 7.6 使用 YAML 配置字体和曲线样式

示例文件：

```text

```

运行：


```text

marker
```

### 7.7 导出绘图使用的 ECDF 点

导出的 CSV 字段为：

```text
scheme,label,x,cdf
```

因此论文图、报告图和图中实际点可以一一对应。

## 9. 建议的正式评估流程

下面是建议流程，不表示这些值已经写入当前三站点 YAML。

### 9.1 第一步：反馈量敏感性

保持其他条件一致，分别运行：

```text
K1 = 1, 2, 4, 8
```

建议让 `oracle_service_beam_top_k` 与 K1 相同，用于公平比较候选搜索空间；另行增加一个更大 oracle K 作为算法上界时，应明确标注其候选集合不同。

### 9.2 第二步：扩大候选 UE 池

建议先比较：

```text
4 UE/sector  = 36 UE
8 UE/sector  = 72 UE
10 UE/sector = 90 UE
```

观察：

- 系统平均吞吐；
- 实际调度 UE 数；
- 调度相似性；
- runtime 和内存；
- 未调度 UE 比例。

### 9.3 第三步：检查 OLLA warmup

建议对相同 drop/seed 比较：

```text
warmup = 0, 50, 100, 200, 500
```

如果正式统计阶段的 ACK rate、TBLER、OLLA offset 和吞吐对 warmup 长度已经不敏感，才可以认为 warmup 基本充分。

### 9.4 第四步：增加蒙特卡洛样本

当前 `1 drop × 10 measured TTIs` 只能用于功能验证。正式数量应根据置信区间和计算资源确定。可以从下面的中等规模开始：

```yaml
system:
  num_drops: 50
  num_tti_per_drop: 500

link_abstraction:
  olla_warmup_tti: 100
```

然后检查不同随机 seed 下的系统吞吐、p05 和调度相似性是否稳定。不能仅凭固定 seed 的一个 drop 得出方案性能结论。

### 9.5 第五步：决定是否需要动态重调度

如果研究目标只是“固定 drop 下的一次 MU 用户分组与波束选择”，当前一次调度模型可以保留。

如果研究目标包括：

- 长期单用户吞吐；
- 5% 用户体验；
- PF 公平性；
- 时变信道下的调度；
- UE 轮转服务；

则应在后续版本中实现 TTI 或周期性重新调度。否则 UE 数大于 36 后，未被首次调度选中的 UE 会在整个 drop 中保持零吞吐。
