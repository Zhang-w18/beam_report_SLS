# v2.8 Gamma 矩阵 GPU 并行计算更新说明

## 1. 结论与现有计算流程核对

当前每个 drop 的调度前测量流程是：

1. 对每个 UE 和调度域内的每个候选 TX beam，计算所有 RX beam 上的接收功率；
2. 对每个 `(UE, TX beam)` 独立选择接收功率最大的 RX beam；
3. 计算服务功率 `S[u,m]`，并在服务 beam `m` 所选的 RX beam 下计算潜在干扰 beam `n` 的功率；
4. 构造：

   $$
   \Gamma_u(m,n)=\frac{S_u(m)}{I_u(m,n)+N_u},\quad m\ne n,
   $$

   对角线为单用户 SNR：

   $$
   \Gamma_u(m,m)=\frac{S_u(m)}{N_u}.
   $$

因此，“先计算所有 TX/RX beam 组合、再为每个 TX beam 选最优 RX beam、最后构造 Gamma”这一理解是正确的。需要注意，最优 RX beam 不是每个 UE 只选一个，而是每个 `(UE, TX beam)` 各选一个。

调度完成后，`beam_sls/link.py::realized_sinr_grid()` 不直接使用调度前的 pairwise Gamma 作为最终 SINR，而是根据每个 scheme 的实际 `schedule.links`、真实频域信道 `H`、已选 TX/RX beam，在每个频点累加所有实际已调度干扰，再执行 EESM、TBLER 和 ACK/NACK 计算。因此调度后的真实干扰仍然包含合并调度集合中的全部发射 beam。

## 2. 多 scheme 是否重复计算 Gamma

不会因 scheme 数量而重复计算。

当前调用关系为：

```text
每个 drop
  ├─ 生成一次信道 H
  ├─ compute_gamma_measurement() 一次
  ├─ make_reports() 从同一个 MeasurementResult 生成所有 feedback scheme 报告
  └─ 各 evaluation case/scheme 分别调度并重算调度后真实干扰
```

因此：

- 同一个 drop 内，`full_gamma`、`baseline`、`topk_conflict_id`、`threshold_conflict_set` 共用一次调度前测量；
- 同一 feedback scheme 配多个调度算法时，也共用该测量；
- 不同 drop 的 UE、信道和 Gamma 不同，必须重新计算；
- coverage heatmap 和 fixed vertical beam CDF 是独立覆盖评估，会另外调用测量函数，不属于某个调度 scheme 的重复计算。

## 3. 哪些计算是多余的

### 3.1 已有的调度域裁剪

主仿真已经通过 `candidate_beam_indices_by_ue` 按调度域裁剪：

- `per_sector_independent`：只测 serving sector；
- `per_site_joint`：只测 UE 所属 site；
- `global`：测全网 beam。

域外 Gamma 不会分配完整三维 dense tensor，而由 `SparseGamma` 以全局 beam index 提供稀疏访问。这部分不是多余计算。

### 3.2 `interference_power_w`

`interference_power_w` 在仓库中没有调度、反馈、链路评估或结果输出消费者。它与 Gamma、服务功率和噪声之间满足：

$$
I_u(m,n)=\frac{S_u(m)}{\Gamma_u(m,n)}-N_u,\quad m\ne n,
$$

因此再保存一份 `[UE,B,B]` tensor 是重复的。v2.8 不再分配或填充该 tensor，`MeasurementResult.interference_power_w` 暂时保留为空数组 `(0,0,0)`，用于降低数据类接口变更范围。外部代码不应再读取该字段。

### 3.3 仍存在但本次没有改变的候选行计算

所有 scheme 生成报告前都必须先获得每个调度域 beam 的 SU 服务质量，才能确定 top-K 服务 beam。因此 TX/RX 扫描不能简单删掉。

当前 Gamma 仍对调度域内所有服务 beam 行计算。严格来说，报告和调度最终只会读取 top-K 服务候选对应的行：

- baseline 不读取非对角 Gamma；
- Top-K conflict 和 threshold conflict 只读取上报服务候选行；
- full-Gamma 调度只会把 oracle top-K 候选作为服务 beam，但需要这些行对所有潜在干扰 beam 的列。

先完成 SU 扫描，再只计算各 scheme 候选行的两阶段实现还能减少计算；但它会改变 `SparseGamma` 的完整行语义，并使 measurement 与 feedback 的 top-K 配置耦合。为满足“最小改动、保持现有结果和接口”的要求，v2.8 没有做这项裁剪。

### 3.4 `measurement.compute_full_gamma`

该历史配置项目前没有被代码读取；现有路径无论它是 `true` 还是 `false`，都会按调度域计算 Gamma。v2.8 没有改变这一旧行为，以免在有限反馈和 full-Gamma 同时运行时出现隐式缺失数据。是否计算 pairwise Gamma 由实际 feedback scheme 自动推导，可作为后续独立优化。

## 4. 哪些中间结果可以复用到调度后

当前已经复用：

- `selected_rx_beam[u,m]`：调度后真实链路使用调度前为服务 beam 选择的 RX beam；
- `service_power_w`、`Gamma`、`noise_power_w`：用于生成所有 scheme 报告和 full-Gamma 调度预测；
- 同一 drop 的信道 `H` 和 beam codebook：所有 scheme 共用。

调度前频率平均后的 pairwise Gamma 不能直接替代调度后真实干扰，原因是：

- 调度后可能同时存在多个干扰 beam，必须在线性功率域求和；
- 链路评估保留每个频点的 SINR，再做 EESM；
- 调度前 Gamma 是单个干扰 beam 的频率平均 pair 信息，不能无损恢复多干扰频率网格。

理论上可以缓存每个候选服务状态下的复数有效信道：

$$
z_{u,m,n}(f)=q_{u,m}^{H}H_{u,n}(f)w_n,
$$

但完整缓存规模约为 `UE × service beam × interferer beam × frequency`，通常比 Gamma 大 `num_freq_points` 倍。较合理的后续方案是只缓存所有 scheme top-K 候选行，或者对重复的 `(UE, service beam, interferer beam)` 按需 memoize。本次没有增加这类大缓存。

## 5. v2.8 代码改动

### 5.1 NumPy/CuPy 双后端

`beam_sls/measurement.py` 的 `compute_gamma_measurement()` 新增：

```python
compute_backend="numpy"  # numpy / cupy / auto
ue_batch_size=0
```

- `numpy`：使用 CPU，保持默认兼容行为；
- `cupy`：必须使用可工作的 CuPy 和 NVIDIA CUDA GPU，初始化失败时直接报错；
- `auto`：优先 CuPy，CuPy 不可用时回退 NumPy并记录原因。

生产服务器上建议使用 `cupy`，以免 CUDA 配置错误时静默回退 CPU。

### 5.2 UE 分组和 batch 并行

UE 先按完全相同的调度域 beam 列表分组。同一 site/sector/domain 的多个 UE 通常会进入同组，再按 `gamma_ue_batch_size` 执行批量：

1. 按 TX unit 读取当前 UE batch 的信道切片；
2. 用 batched `einsum` 同时计算多个 UE、多个 TX beam 的 `Hw`；
3. 一次投影到所有 RX beam；
4. 在 GPU 上执行功率平均、最优 RX beam 选择、服务功率 gather 和 Gamma 广播计算；
5. 只把 `service_power_w`、`selected_rx_beam` 和 Gamma 结果复制回 CPU。

完整 `h_freq` 不会一次复制到显存。显存占用主要由一个 batch 的以下临时量决定：

```text
H slice : batch × F × Nr × Nt（每次一个 TX unit）
Hw      : batch × Bdomain × F × Nr
RX 投影 : batch × R × Bdomain × F
Gamma   : batch × Bdomain × Bdomain
```

### 5.3 TensorFlow/Sionna 共存

当选择 `cupy` 或 `auto` 时，仿真入口会在 TensorFlow 初始化前设置：

```text
TF_FORCE_GPU_ALLOW_GROWTH=true
```

这避免 Sionna/TensorFlow 提前占满全部显存，使 CuPy 可以在同一进程中申请 Gamma batch 所需显存。如果服务器已经显式设置该环境变量，程序不会覆盖。

### 5.4 新配置、CLI 和运行记录

默认配置新增：

```yaml
measurement:
  gamma_backend: numpy
  gamma_ue_batch_size: 0
```

`gamma_ue_batch_size: 0` 表示自动选择：GPU 为 8，CPU 为 1。

CLI 新增：

```text
--gamma-backend numpy|cupy|auto
```

每个 drop 的实际后端、状态、batch 配置和耗时写入：

```text
metrics/gamma_measurement_backend.csv
```

## 6. 服务器依赖

CPU 路径不增加依赖，继续使用项目现有 `numpy`。

GPU 路径新增必需条件：

- NVIDIA GPU 和兼容驱动；
- 与 CUDA 主版本匹配的 CuPy wheel；
- 不需要修改现有 Sionna/TensorFlow 安装，但它们必须与服务器驱动兼容。

CUDA 12.x：

```bash
python -m pip install ".[gpu]"
# 等价的核心新增包：python -m pip install cupy-cuda12x
```

CUDA 13.x：

```bash
python -m pip install ".[gpu-cuda13]"
# 等价的核心新增包：python -m pip install cupy-cuda13x
```

如果环境没有系统 CUDA Toolkit、但驱动满足要求，可按 CuPy 官方方式安装带 CUDA 组件的 extra：

```bash
python -m pip install "cupy-cuda12x[ctk]"
# 或 cupy-cuda13x[ctk]
```

同一个 Python 环境只能安装一个 CuPy 分发包，不要同时安装 `cupy`、`cupy-cuda12x` 和 `cupy-cuda13x`。

安装后检查：

```bash
nvidia-smi
python -c "import cupy as cp; print(cp.__version__); print(cp.cuda.runtime.getDeviceCount()); cp.show_config()"
```

项目也提供一键检查脚本。它会检查基础包、CuPy 分发包冲突、CUDA runtime/driver、可见 GPU、显存，并实际执行一次与 Gamma 路径相同的 complex128 batched `einsum`：

```bash
python scripts/check_gamma_gpu_env.py
```

最后输出 `[READY]` 且退出码为 0，表示可以使用 `measurement.gamma_backend: cupy`；输出 `[NOT READY]` 时，根据前面的 `[FAIL]` 项安装或修复环境。

本实现只使用 CuPy 核心 ndarray、`einsum` 和 CUDA runtime；SciPy、NCCL、cuDNN 不是必需依赖。cuTENSOR 可作为 CuPy 的可选 tensor 运算加速库，但不是运行条件。

## 7. 使用方法

### 7.1 YAML 启用 GPU

```yaml
measurement:
  num_freq_points: 24
  compute_full_gamma: true
  frequency_average: linear_power
  gamma_backend: cupy
  gamma_ue_batch_size: 8
```

然后按原方式运行：

```bash
python -m beam_sls.run \
  --config configs/v2_three_site_global_36ue.yaml \
  --out runs/v2_8_gpu
```

### 7.2 不改 YAML，使用 CLI

```bash
python -m beam_sls.run \
  --config configs/v2_three_site_global_36ue.yaml \
  --out runs/v2_8_gpu \
  --gamma-backend cupy
```

### 7.3 显存不足时

先减小：

```yaml
measurement:
  gamma_backend: cupy
  gamma_ue_batch_size: 4
```

仍不足时继续改为 2 或 1。`1` 仍会在 GPU 上并行 TX beam、RX beam、频点和 Gamma 元素，只是不再同时处理多个 UE。

如果机器与其他任务共享 GPU，可在启动前限制 CuPy 内存池：

```bash
export CUPY_GPU_MEMORY_LIMIT="50%"
```

CuPy 默认使用内存池，因此 `nvidia-smi` 中看到计算结束后仍保留一部分显存是正常的，后续 batch 会复用该内存。

## 8. 兼容性和结果

- 默认仍是 `gamma_backend: numpy`，旧配置不需要修改；
- CPU 和 GPU 都使用 complex128/float64 路径，保持原有精度；
- 不改变 Gamma 定义、RX beam 选择规则、反馈内容、调度器或调度后真实 SINR；
- GPU 浮点规约顺序可能造成极小的末位差异，不应影响正常调度结果；
- `interference_power_w` 改为空数组是唯一有意的数据字段行为变化；仓库内部不读取该字段；
- 本地无 NVIDIA/CuPy 环境时可以完整使用 NumPy路径；服务器 GPU 实际加速比应以 `metrics/gamma_measurement_backend.csv` 的 `elapsed_s` 对相同配置做 CPU/GPU 对照。
