# 修改说明 v2.4

本文记录 v2.4 中新增调度算法和仿真诊断输出的实现方式、统计口径与配置注意事项。这里的口径应与代码一起保留，避免后续分析时对“单用户吞吐”“相同调度结果”或“无干扰上界”产生不同理解。

## 1. 新增调度算法

### 1.1 硬冲突约束 greedy

配置方式：

```yaml
scheduler:
  algorithm: hard_conflict_greedy
```

每个 UE 上报的每个候选服务 beam 都构成一个独立节点：

```text
(ue_id, beam_index)
```

节点权重为该候选链路的单用户 SU rate，单位为 Mbps。SU rate 由候选的 `su_mcs` 通过当前 link-adaptation backend 转换得到。

候选节点 `a=(u_i,b_i)` 与 `b=(u_j,b_j)` 满足下面任意一项时视为冲突：

```text
b_j in conflict_beams(u_i, b_i)
或
b_i in conflict_beams(u_j, b_j)
```

因此冲突判断是双向 OR，而不要求两个 UE 同时上报对方。

每轮从候选池中选择 SU rate 最大的节点。选中 `(u,b)` 后删除：

1. UE `u` 的所有其他候选 beam，因为一个 UE 在同一调度结果中只能选择一个服务 beam；
2. 与 `(u,b)` 冲突的候选二元组；
3. 与 `(u,b)` 使用同一独立 TX unit 的候选节点，用于满足 `use_panel_constraint`。

第二项只删除具体的冲突候选二元组，不删除冲突候选所属的整个 UE。例如 `(UE1, beam1)` 与当前节点冲突时，`(UE1, beam2)` 仍保留在候选池中，后续仍可能被调度。这是该算法最重要的实现语义。

SU rate 相同时，代码依次按 `ue_id` 和 `beam_index` 做确定性 tie-break，保证相同输入得到相同结果。

算法最多选择 `max_mu_order` 个节点。最终 `objective_value` 会按配置的调度目标重新计算；硬冲突选择过程本身的节点排序严格使用 SU rate。

### 1.2 自适应 lambda

可以直接使用算法别名：

```yaml
scheduler:
  algorithm: adaptive_lambda_greedy
  adaptive_lambda_alpha: 0.2
```

也可以在普通 greedy 或 exhaustive 中打开自适应模式：

```yaml
scheduler:
  algorithm: greedy
  conflict_penalty_mode: adaptive
  adaptive_lambda_alpha: 0.2
```

lambda 在每个调度域内独立计算：

```text
lambda = adaptive_lambda_alpha * median(当前调度域所有候选 SU rate)
```

这里的中位数样本是所有 UE 上报的所有候选 `(UE, beam)`，不是只统计每个 UE 的最佳 beam，也不是从最终调度集合计算。SU rate 和 lambda 的单位都是 Mbps。

`per_site_joint` 会逐站点得到不同 lambda，`per_sector_independent` 会逐扇区得到不同 lambda，`global` 只计算一个全网 lambda。

冲突惩罚仅用于 `topk_conflict_id` 和 `threshold_conflict_set`。当前 utility 中的冲突计数保留上报方向：如果只有 A 上报 B，则计数一次；如果 A、B 都上报对方，则计数两次。因此 lambda 是“每个有向冲突上报”的惩罚尺度。

下面字段写入 `metrics/scheduler_stats.csv`，用于确认自适应参数是否正确生效：

```text
conflict_penalty_mode
conflict_penalty_lambda_mbps
adaptive_lambda_alpha
candidate_su_rate_median_mbps
candidate_count_for_lambda
```

## 2. baseline 无波束间干扰参考上界

输出 scheme 名称为：

```text
baseline_no_interference_upper_bound
```

实现步骤如下：

1. 正常运行 baseline 调度器，得到固定的 `(UE, beam)` 调度集合；
2. 不重新选择 UE，也不重新选择 beam；
3. 链路层重新计算真实信道上的逐频点 SINR，但分母只保留热噪声，强制跳过所有其他已调度 beam 的干扰功率；
4. 使用与正常 baseline 相同的 predicted MCS、EESM、BLER、ACK、TBS 和 OLLA 流程计算实际 goodput；
5. 复制正常 baseline 链路评估前的随机数状态，使两者使用相同的 ACK 随机数序列，减少随机采样对比较的影响。

因此它回答的是：“baseline 已经选定的调度集合，在波束间干扰为零时能够得到多少链路层吞吐”。它不是重新搜索所有可能调度集合得到的数学全局最优值，而是 baseline 调度结果的无干扰诊断上界。

该结果写入 `metrics/link_tti.csv` 和 `metrics/summary.csv`，并进入吞吐/SINR图。可用字段 `ratio_to_baseline_no_interference_upper_bound` 查看各 scheme 相对该参考值的比例。

配置开关：

```yaml
analysis:
  baseline_no_interference_upper_bound: true
```

## 3. 单用户吞吐 CDF 和 5% 边缘吞吐

### 3.1 样本定义

单用户吞吐样本的 key 是：

```text
(scheme, drop, ue_id)
```

对每个样本，将该 UE 在当前 drop 内所有 TTI 的 `goodput_mbps` 求和，再除以配置的 `num_tti_per_drop`：

```text
avg_ue_goodput = sum(goodput_mbps over all TTIs) / num_tti_per_drop
```

这里除数始终是该 drop 的全部 TTI 数量，而不是该 UE 实际被调度的 TTI 数量。

### 3.2 未调度 UE 的处理

每个 drop 中的所有 UE 都会为每个 scheme 生成一条记录。完全没有被调度的 UE 也必须进入统计，其吞吐记为 0 Mbps，`scheduled` 字段记为 0。

这是为了避免只统计已调度 UE 带来的幸存者偏差。如果只保留有 `link_tti.csv` 记录的 UE，CDF 和 5% 边缘吞吐会被明显高估。

明细输出：

```text
metrics/ue_goodput.csv
```

CDF 输出：

```text
figures/ue_goodput_cdf.png
```

### 3.3 5% 边缘吞吐和增益

`p05_ue_goodput_mbps` 是上述全部 `(drop, UE)` 样本的 5% percentile，使用 NumPy percentile 计算，未调度 UE 的零值包含在内。

相对 baseline 同时记录：

```text
p05_gain_over_baseline_mbps = scheme_p05 - baseline_p05
p05_gain_over_baseline = (scheme_p05 - baseline_p05) / baseline_p05
```

如果 baseline 的 5% 吞吐为零，相对百分比没有数学意义，因此非 baseline scheme 的相对字段留空；绝对 Mbps 增益仍正常输出。baseline 自身的相对增益记为 0。

当前仿真在每个 drop 内生成一次 schedule，并在该 drop 的所有 TTI 中复用。因此当 UE 数远大于 MU order 时，大量 UE 会在整个 drop 内保持零吞吐，p05 可能为零。这是当前调度时序模型的真实结果，不应在后处理中删除这些 UE。

## 4. 调度相似度

每个 scheme 在每个 drop 的调度结果转换成集合：

```text
S = {(ue_id, beam_index), ...}
```

必须同时具有相同 UE 和相同全局 `beam_index` 才算相同调度二元组。只调度了相同 UE、但 beam 不同，不算相同。

对任意两个 scheme A、B，主要指标 Jaccard 相似度为：

```text
J(A,B) = |A ∩ B| / |A ∪ B|
```

如果两个集合都为空，Jaccard 定义为 1。

同时输出以下辅助指标：

```text
same_pair_ratio_over_max_size = |A ∩ B| / max(|A|, |B|)
overlap_coefficient           = |A ∩ B| / min(|A|, |B|)
exact_schedule_match          = 1 if A == B else 0
```

`schedule_similarity_by_drop.csv` 保存逐 drop 结果。`schedule_similarity.csv` 保存跨 drop 汇总：

- `mean_jaccard_similarity`：先算每个 drop 的 Jaccard，再对 drop 做等权平均；
- `micro_jaccard_similarity`：所有 drop 的交集数量之和除以所有 drop 的并集数量之和；
- `exact_schedule_match_ratio`：两个 scheme 完全相同的 drop 比例。

## 5. UE 上报 SU-SNR 统计

### 5.1 所有候选上报样本

`metrics/su_snr_samples.csv` 中，每个 UE 上报的每个候选 beam 都生成一个独立样本。同一个 UE 上报两个 `su_snr_db` 时会计入两个样本，不先做 UE 平均。

字段包括：

```text
drop, scheme, ue_id, candidate_rank, beam_index, beam_id, su_snr_db
```

所有候选样本的平均值、p05 和 p50 写入 `metrics/su_snr_summary.csv`。CDF 为：

```text
figures/reported_su_snr_cdf.png
```

### 5.2 每个 UE 的最大 SU-SNR

对每个 `(drop, scheme, ue_id)`，在该 UE 上报的所有候选 beam 中取最大 `su_snr_db`，每个 UE 最终只贡献一个样本。

明细和 CDF 为：

```text
metrics/su_snr_max_per_ue.csv
figures/reported_max_su_snr_per_ue_cdf.png
```

该分布主要用于观察小区内 UE 的远近和最佳可用链路质量。由于 `full_gamma` 与其他反馈 scheme 的候选数量可能不同，统计按 scheme 分开输出，不跨 scheme 重复混合。

## 6. 三站点全局 36 用户并发配置

仓库原有 `configs/v2_three_site_triangle.yaml`，但它使用 `per_site_joint`，每个站点单独形成调度域，并不是三个站点组成一个全局调度问题。

新增配置：

```text
configs/v2_three_site_global_36ue.yaml
```

拓扑与并发关系为：

```text
3 sites
× 3 sectors/TRPs per site
× 4 TXRUs per TRP
= 36 independent TX units
```

在当前数据模型中，一个 sector/cell 对应一个 TRP，因此配置使用：

```yaml
topology:
  num_sites: 3
  sectors_per_site: 3

trp:
  num_trps_per_sector: 1
```

这里不能把 `num_trps_per_sector` 再设成 3，否则会变成每个 sector 内 3 个 TRP，总数将错误地扩大到 27 个 TRP。

调度域设置为 `global`。当前 RF cap 的自动解析值是单 sector/TRP 的 4，而全局域需要聚合 9 个 TRP，所以该 YAML 显式使用：

```yaml
scheduler:
  domain_mode: global
  max_mu_order: 36
  cap_mu_order_by_rf: false
  use_panel_constraint: true
```

关闭单 TRP cap 不会允许超过真实硬件资源：全局调度仍用 `(cell, site, TXRU)` 对应的 `panel_key` 约束每个独立 TX unit 同时只能选择一个 beam，同时 `max_mu_order: 36` 给出全网总上限。

配置每个 sector 放置 4 个 UE，全网正好 36 个 UE；每个 UE 上报 56 个候选 beam，使其能够覆盖服务 TRP 的 4 个 TXRU，而不是因为 `k1=2` 只覆盖少数 TXRU。一个服务 TRP 自身有 `4×8=32` 个 beam，但全局 SU-rate 排序可能把邻区 beam 排进前 32，所以候选数等于 32 并不等于覆盖了 4 个服务 TXRU。

候选数经过了两层验证：直接使用主 seed 构造的场景中，`k1=32` 只能调度 34 个 UE，`k1=40` 可以达到 36；但 `run_simulation()` 会从主 RNG 再生成每个 drop 的 seed，在这条正式运行路径中 `k1=40` 和 48 只能调度 35 个 UE，`k1=56` 才得到 36 个 UE、36 个唯一 TX unit。因此最终 YAML 使用 56，并固定 `num_drops: 1` 和 `random_seed: 20260714`。如果修改随机种子、drop 数、beam 数或天线参数，应重新检查 `metrics/schedules.csv` 的 `num_scheduled`，必要时继续增加 `k1`。

baseline greedy 的 SU utility 始终为正，因此在候选和 TX unit 约束可满足时会填满 36 个位置。冲突感知 scheme 可能因为冲突惩罚或真实干扰效用主动少调度，这属于算法结果。

为了使 36-TX-unit 全局仿真能够在普通工作站上实际运行，该 YAML 使用较小但仍保持 4 TXRU/TRP 的 64-AE TRP array、4-AE UE array、每 TXRU 8 个候选 beam和 2 个频率采样点。默认 1024-AE array 在当前显式保存 `H[UE, TX unit, frequency, RX antenna, TX antenna]` 的实现下会消耗大量内存。

### 6.1 标准运行命令

在项目根目录执行：

```bash
cd /Users/zhangwei/Downloads/sionna_sls_beam_mgmt_v2_4

python -m beam_sls.run \
  --config configs/v2_three_site_global_36ue.yaml \
  --out runs/v2_4_three_site_global_36ue
```

如果服务器使用专门的 Sionna Python 环境，请将 `python` 替换为对应解释器，例如：

### 6.2 运行硬冲突 greedy

命令行可以覆盖 YAML 中的调度算法：

```bash
python -m beam_sls.run \
  --config configs/v2_three_site_global_36ue.yaml \
  --out runs/v2_4_three_site_global_hard_conflict \
  --algorithm hard_conflict_greedy
```

硬冲突算法可能因为候选冲突而少于 36 个用户，这是硬约束删除的正常结果。

### 6.3 运行自适应 lambda greedy

`adaptive_lambda_alpha` 已在 YAML 中设置为 0.2，可以直接运行算法别名：

```bash
python -m beam_sls.run \
  --config configs/v2_three_site_global_36ue.yaml \
  --out runs/v2_4_three_site_global_adaptive_lambda \
  --algorithm adaptive_lambda_greedy
```

如果需要测试其他 alpha，应复制该 YAML 或直接修改：

```yaml
scheduler:
  algorithm: adaptive_lambda_greedy
  adaptive_lambda_alpha: 0.5
```
