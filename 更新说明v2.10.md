# v2.10 调度器可观测性与增量向量化更新说明

## 1. 为什么要修改

v2.9 已经把调度前的 ILLA 范围缩小到各 scheme 实际会上报的 top-k 服务波束，但 full-Gamma greedy 在每一个 trial 中仍会从头执行以下工作：

1. 从头扫描当前调度集合检查 panel 约束；
2. 对 trial 中每条链路重新遍历全部其它链路并查询 Gamma；
3. 对每条链路单独调用一次 Sionna ILLA；
4. 再单独调用一次 PHYAbstraction 判断所选 MCS 是否 outage；
5. 重新计算整个 trial 的目标函数。

对于三站点全局配置中的 45 UE、每 UE 4 个候选、最大 MU 阶数 18，在不考虑 panel 剪枝的上界下，一个 full-Gamma case 最多会产生：

- 2,628 次 trial；
- 23,028 次 ILLA；
- 23,028 次 PHYAbstraction；
- 248,064 次 Gamma 元素访问。

这些调用的数据量很小，却会反复触发 Python、TensorFlow 和 NumPy 之间的切换及同步。因此程序并非死锁，而是可能长时间停留在第一个 `full_gamma__greedy` case，看起来像调度器卡住。

v2.10 的目标是：

- 让 case 和 greedy 每一轮的进度可见；
- 消除调度热点循环中的 TensorFlow 小调用；
- 将 full-Gamma trial 从重复重建改为增量干扰更新；
- 将同一轮全部候选交给 NumPy 批量计算；
- 对有限反馈 greedy 使用静态节点权重和增量冲突代价；
- 保留 v2.9 贪心路径，便于结果回归和故障排查；
- 不改变调度后的真实信道、EESM、ILLA、TBLER、ACK/NACK 和吞吐评估路径。

## 2. 修改了什么

### 2.1 case 级和 greedy 轮次级计时

仿真现在会输出类似：

```text
[drop 1/10][case 1/6] full_gamma__greedy started
[drop 1/10][case 1/6] full_gamma__greedy q=1/18, candidates=180, elapsed=0.012s
[drop 1/10][case 1/6] full_gamma__greedy q=2/18, candidates=166, elapsed=0.014s
[drop 1/10][case 1/6] full_gamma__greedy scheduler finished, elapsed=0.031s
[drop 1/10][case 1/6] full_gamma__greedy finished, link_eval_elapsed=0.420s
```

如果使用 per-site 或 per-sector 独立调度，轮次日志还会带 `domain=<id>`。

新增输出：

- `metrics/scheduler_iterations.csv`：每个 drop、case、domain、q 的剩余候选数、实际评估候选数、panel 剪枝数和累计耗时；
- `metrics/runtime_phases.csv`：查表初始化、反馈生成、scheduler 和实际链路评估的分阶段耗时及调用计数；
- `metrics/scheduler_stats.csv`：增加 `scheduler_elapsed_s`、调度查表查询数、ILLA/PHY 后端调用数、查表边界回退数和实际链路评估耗时。

重要计数字段包括：

- `evaluated_assignment_count`：调度器实际评估的 trial 数；
- `panel_pruned_count`：因 panel 约束跳过的候选数；
- `scheduler_lookup_queries`：调度阶段通过 NumPy 门限表完成的 SINR 决策数；
- `scheduler_illa_invocations` / `scheduler_illa_items`：调度阶段真正进入 Sionna ILLA 的调用批次数和标量元素数；
- `scheduler_phy_abstraction_invocations` / `scheduler_phy_abstraction_items`：调度阶段真正进入 PHYAbstraction 的调用批次数和元素数；
- `scheduler_lookup_boundary_fallback_items`：位于决策门限保护带内、为保证一致性而回退原始后端的元素数；
- `link_evaluation_illa_*` 和 `link_evaluation_phy_abstraction_*`：实际 TTI 链路评估阶段的原始 Sionna 调用数。

### 2.2 调度专用 MCS/outage/rate 决策查表

v2.10 在初始化阶段建立独立的 `SchedulerLinkLookup`。它只包装调度和报告生成所用的 link adapter，不替换 `run_tti_loop()` 中的原始 adapter。

构建过程如下：

1. 在配置 SINR 范围内批量调用一次或少量几次原始后端；
2. 同时取得每个 SINR 的 selected MCS 和 outage；
3. 找出 `(MCS, outage)` 发生变化的位置；
4. 对每个变化位置调用原始后端二分细化；
5. 保存几十个分段门限，而不是在热点循环中保存或查询完整 TensorFlow 表；
6. 用额外采样点重新调用原始后端验证门限表；验证失败时直接报错；
7. 正常查询使用 `np.searchsorted`；位于门限保护带内或超出扫描范围的值自动回退原始后端。

查表同时返回：

- `selected_mcs`；
- `outage`；
- outage 后的有效 `rate_mbps`。

因此调度热点循环不会出现“ILLA 已查表，但 outage 仍逐条调用 PHY”的半优化状态。

实际 TTI 链路评估仍使用原始 Sionna：

```text
真实频域信道 H
  -> 合并调度集合的真实干扰
  -> EESM
  -> OLLA 后 SINR
  -> 原始 Sionna ILLA
  -> 原始 PHYAbstraction TBLER
  -> ACK/NACK 与 goodput
```

### 2.3 full-Gamma 紧凑节点和干扰矩阵

每个 `(UE, service beam)` 被转换为一个紧凑 node。对于 45 UE、每 UE 4 个候选，一共只有 180 个 node。

初始化时一次性建立：

\[
I_{ij}=\max\left(0,\frac{S_i}{\max(\Gamma_i(m_i,m_j),10^{-30})}-N_i\right).
\]

这里的 `max(0, ...)` 和 `10^-30` 与 v2.9 `_evaluate_assignments()` 的数值保护保持一致。

180×180 的 `float64` 干扰矩阵只有约 0.25 MiB。对于 `SparseGamma`，这一步还会把热点循环中的稀疏字典访问转换为连续 NumPy 数组访问。

### 2.4 full-Gamma 增量分母

调度器维护：

\[
D_i=N_i+\sum_{j\in S,j\ne i}I_{ij}.
\]

尝试加入候选 `x` 时：

- 已选链路 `i` 使用 `D_i + I_ix`；
- 新链路 `x` 使用 `N_x + sum(I_xi)`；
- 选中 `x` 后，只做一次向量加法更新当前分母。

因此不再为每个 trial 重复执行 q×q 次 Gamma 查询。

### 2.5 同一轮候选批量计算

第 q 轮把全部合法候选一次构造成：

- 已选链路加入各候选后的 SINR：`[q, C]`；
- 新候选自身 SINR：`[C]`；
- 合并后的 trial rate 和 utility：`[q+1, C]` 和 `[C]`。

随后一次 `argmax` 选择最佳候选。当前规模约为 180×18，默认使用 CPU NumPy，不引入 CUDA kernel launch 和同步开销。

原 full-Gamma Gamma 访问复杂度近似为 `O(UKQ^3)`，增量向量化后为 `O(UKQ^2)`，另加每个调度域一次 `O(C^2)` 干扰矩阵构建。

### 2.6 有限反馈增量效用

baseline、top-k conflict 和 threshold conflict 的 SU rate 在一次 schedule 调用内保持不变。v2.10 预先缓存：

- node 对应的 UE、beam、panel key；
- standalone weighted rate；
- 有向冲突邻接矩阵。

加入候选 `x` 的边际效用为：

\[
\Delta U_x=w_xR_x-\lambda\sum_{i\in S}(A_{xi}+A_{ix}).
\]

每选中一个节点，只需用邻接矩阵的一行和一列更新全部剩余节点的累计冲突代价。最终选定调度集合后，仍调用一次 `_evaluate_assignments()` 生成统一的输出字段。

`hard_conflict_greedy` 原本已经采用候选池删除方式，v2.10 保留其语义并补充轮次计时、候选缓存和统一统计。

### 2.7 panel 与候选索引缓存

普通 greedy 现在和 exhaustive 一样维护 `used_panel_keys`。候选是否合法只需要 O(1) 集合查询，不再为每个 trial 扫描完整已选集合。

`UEReport` 在创建时同时建立 `beam_index -> ServiceCandidate` 字典，`candidate_by_beam()` 不再线性扫描候选列表。

### 2.8 已调度 UE 的 SU 吞吐 CDF

仿真现在会对每个 drop、每个 evaluation case 的最终已调度 UE，读取其实际选中
service beam 对应的 standalone SNR、SU MCS 和 SU outage，并按当前 PDSCH 资源配置
换算无共调度干扰的 SU 吞吐。SU outage 样本按 0 Mbps 计入 CDF。

这个统计不使用 full-Gamma 的 MU predicted rate，也不使用调度后的实际 TTI goodput，
因此可以直接观察不同调度方案最终选中了怎样的单用户吞吐人群。

新增输出：

- `metrics/scheduled_ue_su_throughput.csv`：逐 drop/case/UE 的所选 beam、SU SNR、SU MCS、outage 和 SU throughput；
- `metrics/scheduled_ue_su_throughput_summary.csv`：各 case 的样本数、均值、P05、P50、P95 和 SU outage 比例；
- `figures/scheduled_ue_su_throughput_cdf.png`：各 evaluation case 的已调度 UE SU 吞吐 CDF。

### 2.9 相同调度的 paired-case 调试

`analysis.paired_case_debug` 可指定需要逐项核对的 evaluation case 对。启用后：

- `link_tti.csv` 增加 `ack_random_uniform` 和 `link_position`；
- 终端输出少量 `[paired-debug]` 行；
- 同样的内容写入 `metrics/paired_case_debug.txt`。

检查依次覆盖 schedule 集合和顺序、realized SINR、OLLA、MCS 选择 SINR、actual MCS、
TBLER、ACK 随机数、ACK 和 goodput。若存在差异，只额外输出第一处分叉的
drop/TTI/UE 和两侧取值，便于直接复制回传。

## 3. 结果兼容与参考路径

默认启用 v2.10 优化：

```yaml
scheduler:
  optimized_greedy: true
```

如需逐项对照 v2.9，可以设置：

```yaml
scheduler:
  optimized_greedy: false
```

这会恢复普通/adaptive greedy 的 v2.9 trial 重算路径，但保留 v2.10 的 case/round 日志与统计。调度后的真实链路评估无论该开关为何值都保持不变。

v2.10 保留了候选遍历顺序、严格 `delta > 0` 停止条件、有向冲突计数、PF 权重、outage 零速率和 Gamma 数值裁剪。由于向量求和和 Python 标量求和的浮点结合顺序可能不同，在极端的完全并列或恰好落在 MCS 门限的构造输入上，理论上仍可能出现不同但同效用的选择；门限保护带和参考开关用于定位这类情况。

## 4. 怎么使用

### 4.1 默认使用

原有命令无需修改。只要使用包含 v2.10 默认配置的 `load_config()` 或配置文件，greedy 会自动走优化路径。

例如：

```powershell
python -m beam_sls.run `
  --config configs/v2_7_three_site_global_18ue_distance_range_v4_5ue_eval_matrix.yaml `
  --out runs/v2_10_three_site_global
```

启动时应看到：

```text
[init] scheduler link lookup ready, regions=..., elapsed=...s
```

随后每个 case 会持续输出 q 级进度，不会再只有一条笼统的 “scheduling 6 evaluation cases”。

### 4.2 查表配置

默认配置为：

```yaml
scheduler:
  link_lookup:
    enabled: true
    sinr_min_db: -40.0
    sinr_max_db: 80.0
    scan_step_db: 0.01
    refine_tolerance_db: 1.0e-6
    boundary_guard_db: 1.0e-5
    validation_points: 257
```

参数含义：

- `sinr_min_db/max_db`：初始化时扫描决策变化的范围；超范围查询不会截断，而会回退原始后端；
- `scan_step_db`：发现候选跳变区间的扫描步长；不是最终查询量化步长；
- `refine_tolerance_db`：二分细化停止精度；
- `boundary_guard_db`：查询距离门限小于该值时使用原始后端；
- `validation_points`：构建后与原始后端对照的采样点数。

如果需要完全禁用调度查表：

```yaml
scheduler:
  link_lookup:
    enabled: false
```

此时增量/向量化调度结构仍然生效，但没有 `map_sinr_lin()` 的自定义 adapter 会逐元素调用原始接口，性能会明显下降。标准 Sionna 和 fallback adapter 均支持批量映射。

### 4.3 判断是否仍有后端小调用

查看：

```text
metrics/scheduler_stats.csv
metrics/runtime_phases.csv
```

正常情况下：

- `scheduler_lookup_queries` 较大，这是快速 NumPy 查询数；
- `scheduler_illa_invocations` 和 `scheduler_phy_abstraction_invocations` 接近 0；
- 若出现少量调用，通常来自门限保护带或扫描范围外回退；
- `link_evaluation_illa_*` 和 `link_evaluation_phy_abstraction_*` 仍为非零，这是保留的真实链路评估。

若 `scheduler_lookup_out_of_range_items` 较多，应根据实际 SINR 范围扩大 `sinr_min_db/max_db`，而不是关闭保护逻辑。

### 4.4 定位慢 case

先比较 `runtime_phases.csv` 的 `scheduler` 与 `link_evaluation`：

- scheduler 慢：查看 `scheduler_iterations.csv` 中哪一轮候选多、耗时突增，以及是否发生大量后端回退；
- link evaluation 慢：说明耗时来自每 TTI 的真实 ILLA/PHY，而非 greedy；
- feedback generation 慢：查看 top-k 报告候选数量和反馈阶段调用计数；
- Gamma measurement 慢：继续查看原有 `gamma_measurement_backend.csv`，这与 v2.10 scheduler 优化是不同阶段。

## 5. 测试与验证

v2.10 增加回归测试，验证：

- 查表在 MCS/outage 门限两侧与原始 adapter 一致；
- 查表范围外自动回退原始 adapter；
- 有限反馈增量 greedy 与 v2.9 参考 greedy 选择相同 `(UE, beam)` 并得到相同目标值；
- full-Gamma 增量向量化 greedy 与 v2.9 参考 greedy 选择相同 `(UE, beam)` 并得到相同目标值；
- 新增 scheduler iteration/runtime phase 文件能由完整 smoke simulation 生成；
- 原有 Gamma、反馈、hard conflict、outage、OLLA 和真实链路评估测试继续通过。

当前本地回归结果：36 项测试全部通过。

## 6. 建议的上线检查

第一次在服务器运行时，建议先用 1 drop：

1. 保持 `optimized_greedy: true` 跑一次并保存输出；
2. 用相同 seed 设置 `optimized_greedy: false` 再跑一次；
3. 对比 `metrics/schedules.csv`、`scheduler_stats.csv` 和最终 summary；
4. 确认优化路径的 `scheduler_illa_invocations` 接近 0；
5. 确认实际链路阶段仍有原始 ILLA/PHY 调用；
6. 再扩大到正式 drops/TTIs。

如果两个路径在完全并列候选上选择了不同但同效用的 beam，可根据 `scheduler_iterations.csv` 定位对应 q；如果目标值、outage 或 MCS 不一致，则应保留两次输出并暂时使用 `optimized_greedy: false` 进行复现。
