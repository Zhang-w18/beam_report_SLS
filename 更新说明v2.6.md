# Evaluation case 二维配置改造说明

## 1. 本次修改

仿真现在支持以“上报方式 × 调度算法”的二维矩阵定义 evaluation case。同一种上报方式可以复用同一份 UE report，分别运行多个调度算法。

每个矩阵单元展开为唯一 case：

```text
<feedback_scheme>__<algorithm>
```

例如 `topk_conflict_id__hard_conflict_greedy`。

主要改动包括：

- 新增 `beam_sls/evaluation.py`，负责矩阵展开、case ID、参考 case 和合法性校验。
- `schedule()` 支持显式传入算法，分站点/分扇区调度时也会向下传递该算法。
- report 按唯一上报方式生成一次；调度、PF、OLLA 和调度相似度按 `case_id` 隔离。
- `link_tti.csv`、`schedules.csv`、`scheduler_stats.csv`、`summary.csv` 等结果增加或保留 `case_id`、`feedback_scheme`、`algorithm` 信息。
- 自动生成的调度后 CDF 会把不同 case 画在同一张图中。
- 离线 `scripts/plot_cdf.py` 支持 `--cases`、`--list-cases` 和 YAML 中的 `case_id`；读取旧 run 时仍回退到 `scheme`。
- reported SU-SNR 等上报前指标仍按上报方式分组，不为不同算法重复画相同曲线。

## 2. 合法组合

| 上报方式 | greedy | exhaustive | hard_conflict_greedy | adaptive_lambda_greedy |
|---|---:|---:|---:|---:|
| full_gamma | 允许 | 允许 | 禁止 | 禁止 |
| baseline | 允许 | 允许 | 禁止 | 禁止 |
| topk_conflict_id | 允许 | 允许 | 允许 | 允许 |
| threshold_conflict_set | 允许 | 允许 | 允许 | 允许 |

禁止规则由代码中的能力表控制。非法组合会在信道生成前报错，例如 full-Gamma 不提供 `conflict_ids`，不能使用 hard-conflict greedy。

## 3. 矩阵配置

推荐的紧凑形式：

```yaml
evaluation:
  matrix:
    full_gamma: [greedy]
    baseline: [greedy]
    topk_conflict_id: [greedy, hard_conflict_greedy]
    threshold_conflict_set: [hard_conflict_greedy]
  references:
    oracle: full_gamma__greedy
    baseline: baseline__greedy
```

也可以使用二维 true/false 表：

```yaml
evaluation:
  matrix:
    full_gamma:
      greedy: true
      exhaustive: false
    topk_conflict_id:
      greedy: true
      hard_conflict_greedy: true
```

`references` 用于确定 `oracle_ratio` 和 `gain_over_baseline` 的分母。当同一种参考上报只有一个 case 时可以自动推断；有多个时建议明确配置。

未配置 `evaluation.matrix` 时，旧的 `feedback.schemes + scheduler.algorithm` 行为保留，旧 run 的指标名称也不变。旧式 `scheduler.algorithm_by_scheme` 仍可用于一对一映射。

## 4. 运行本次新增配置

新增配置文件：

```text
configs/v2_three_site_global_36ue_fixed_v8_5ue_eval_matrix.yaml
```

运行：

```powershell
python -m beam_sls.run `
  --config configs/v2_three_site_global_36ue_fixed_v8_5ue_eval_matrix.yaml `
  --out runs/three_site_fixed_v8_5ue_matrix
```

该配置在 `v2_three_site_global_36ue.yaml` 基础上修改为：

- TX `vertical_beam_mode: fixed`
- TX `fixed_v_index: 8`
- `num_ut_per_sector: 5`，全网共 45 UE，竞争 36 个 TX unit
- `service_beam_top_k1: 4`
- `oracle_service_beam_top_k: 4`
- `olla_step_db: 0.5`
- `olla_warmup_tti: 100`
- full-Gamma、baseline 使用 greedy
- Top-K conflict ID、threshold conflict set 使用 hard-conflict greedy

注意：该基础配置的 TX 每面板垂直 DFT 维数为 4，当前代码对固定索引取模，因此配置值 8 实际对应 `v_index=0`。如果实验意图是物理上的第 8 个不同垂直波束，需要同时扩大垂直阵列/码本维度，不能只改 `fixed_v_index`。

## 5. OLLA warmup 建议

本配置使用 100 TTI warmup。目标 BLER 为 10% 且 `olla_step_db=0.5` 时：

- ACK 使 backoff 减少 0.5 dB；
- NACK 使 backoff 增加 4.5 dB。

0.5 dB 步长收敛快，但稳态抖动也明显。100 TTI 通常足以消除从 0 dB 初值开始的主要瞬态，并与基础配置保持一致。正式统计前建议查看 `link_tti.csv` 的 `olla_offset_db` 和 ACK/TBLER；如果仍有明显漂移，可提高到 200 TTI。由于当前每个 drop 重新初始化 OLLA，warmup 是“每个 drop 100 TTI”。

## 6. 自定义 CDF

列出可画的 case：

```powershell
python scripts/plot_cdf.py runs/three_site_fixed_v8_5ue_matrix `
  --metric ue_goodput --list-cases
```

命令行选择多条曲线：

```powershell
python scripts/plot_cdf.py runs/three_site_fixed_v8_5ue_matrix `
  --metric ue_goodput `
  --cases full_gamma__greedy baseline__greedy `
          topk_conflict_id__hard_conflict_greedy `
          threshold_conflict_set__hard_conflict_greedy
```

使用样式配置：

```powershell
python scripts/plot_cdf.py runs/three_site_fixed_v8_5ue_matrix `
  --config configs/cdf_evaluation_cases_example.yaml
```

调度后指标（goodput、effective SINR、TBLER、OLLA）按 `case_id` 分组；reported SU-SNR 指标按 `feedback_scheme` 分组。
