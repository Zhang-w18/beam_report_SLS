# v2.9 服务波束筛选与链路抽象更新说明

## 1. 为什么修改

v2.8 的调度前测量顺序存在一处不必要的计算：`compute_gamma_measurement()` 在得到所有候选波束的 SU-SNR 后，立即对调度域内每个候选波束调用链路抽象，逐个选择 MCS 并判断 outage；但后续反馈和调度只会使用 UE 上报的 top-k 服务波束。若每个 UE 的调度域有 `B` 个波束、实际只上报 `k1=4` 个波束，原流程会对 `B` 个波束做 ILLA，而不是只处理最终候选的 4 个。

服务波束的排序依据本来就是无干扰接收功率或 SU-SNR，MCS/outage 不是选 top-k 的前置条件。因此应先用 SU-SNR 完成波束筛选，再对入选波束做链路抽象。

此外，调度后的链路评估虽然已经根据真实频域信道累加实际已调度干扰并计算 EESM SINR，但旧代码在选择 `actual_mcs` 时仍使用调度器的预测 SINR/MCS。v2.9 改为按照调度后实际干扰 SINR再次调用链路抽象，使实际 MCS、TBLER 和吞吐来自同一条实际链路状态。

## 2. 修改后的流程

每个 drop 按以下顺序执行：

1. 对每个 UE 和调度域内所有候选 TX beam 扫描 RX beam，得到无干扰服务功率、最优 RX beam 和 SU-SNR。
2. 每个 UE 仅按 SU-SNR 排序，选择最强的 `k1` 个服务波束；默认 `k1=4`。
3. 只对实际会进入报告的 `(UE, service beam)` 调用 ILLA，得到 SU-CQI/MCS 和 SU-outage。多个 feedback scheme 使用同一候选时只计算一次。
4. 用这些候选生成 baseline、冲突反馈和 full-Gamma 报告并执行调度。
5. 对最终被调度的波束组合，使用真实频域信道和已选 TX/RX beam，在每个频点累加所有已调度干扰，经过 EESM 得到实际有效 SINR。
6. 使用实际有效 SINR（启用 OLLA 时先减去 OLLA backoff）再次选择实际 MCS，再计算 TBLER、ACK/NACK、TBS 和吞吐。

调度前预测值仍保存在 `predicted_sinr_db`、`predicted_mcs` 和 `predicted_rate_mbps` 中；调度后结果写入 `actual_mcs`、`effective_sinr_db`、`tbler` 和 `goodput_mbps`，便于比较预测与实际结果。

## 3. 代码如何修改

### 3.1 测量阶段不再调用 ILLA

`beam_sls/measurement.py::compute_gamma_measurement()` 现在只负责物理测量和 SU-SNR 计算，不再接收 `link_adapter`，也不再遍历所有候选波束选择 MCS/outage。

`MeasurementResult.su_mcs` 仍保留以兼容现有数据结构，但未入选波束使用 `-1` 表示尚未执行链路适配；`su_outage` 只对已入选波束有意义。

### 3.2 报告阶段先筛选、后适配

`beam_sls/feedback.py::make_reports()` 新增 `link_adapter` 参数。它先为所有 scheme 计算 top-k 服务波束，再取候选并集并逐一执行：

```text
select_mcs_from_sinr_lin(SU-SNR)
is_outage_from_sinr_lin(SU-SNR, selected_mcs)
```

因此，当 `service_beam_top_k1=4` 且 `oracle_service_beam_top_k=4` 时，每个 UE 每个 drop 最多只对 4 个调度前服务候选执行一次 MCS/outage 判定，而不会按 scheme 重复，也不会对其余候选波束执行。

为保留 full-Gamma 上界实验能力，`full_gamma` 仍读取 `oracle_service_beam_top_k`。若它大于 `service_beam_top_k1`，ILLA 会处理两类报告候选的并集，其数量等于两者中较大的 top-k。要严格保持每 UE 仅 4 个，请把两个参数都设为 4。

### 3.3 调度后按实际 SINR重新适配

`beam_sls/link.py::run_tti_loop()` 现在使用 `realized_sinr_grid()` 和 EESM 得到的 `effective_sinr_db` 选择 `actual_mcs`：

```text
mcs_selection_sinr_db = effective_sinr_db - olla_offset_db
actual_mcs = ILLA(mcs_selection_sinr_db)
tbler = PHYAbstraction(effective_sinr_db, actual_mcs)
```

关闭 OLLA 时 offset 为零，仍然根据实际有效 SINR选择 MCS，不再直接复制 `predicted_mcs`。

## 4. 计算量变化

设每个 UE 调度域内有 `B` 个候选波束，所有 scheme 的服务候选并集大小为 `K`：

- 修改前：调度前 ILLA/MCS/outage 调用规模为 `UE × B`；
- 修改后：调度前调用规模为 `UE × K`；
- 常用配置 `k1=oracle_top_k=4` 时，`K=4`。

全波束接收功率和 SU-SNR 扫描仍然保留，因为它们是正确选择 top-k 的必要输入。Gamma、冲突集合以及 full-Gamma 调度语义没有改变。

## 5. 怎么使用

配置反馈候选数：

```yaml
feedback:
  service_beam_top_k1: 4
  oracle_service_beam_top_k: 4
```

默认配置和单小区/单站点示例配置已经更新为 4。对于专门扩大 oracle 搜索空间的实验，可单独增大 `oracle_service_beam_top_k`，但这会相应增加 full-Gamma 候选的 ILLA 调用数。

运行方式不变，例如：

```powershell
python -m beam_sls.run --config configs/v2_one_site_three_sector.yaml --output outputs/v2_9_run
```

主要输出字段：

- `metrics/su_snr_samples.csv`：top-k 候选的 `su_snr_db`、`su_mcs`、`su_outage`；
- `metrics/schedules.csv`：调度器预测的 SINR/MCS/outage；
- `metrics/link_tti.csv`：调度后实际 `effective_sinr_db`、`actual_mcs`、`tbler`、ACK 和吞吐。

## 6. 验证

v2.9 增加回归测试，验证以下行为：

- 6 个候选波束、`k1=4` 时，只对 SU-SNR 最强的 4 个波束调用 MCS/outage；
- 四种 feedback scheme 共用候选时不会重复调用 ILLA；
- 未入选波束的 `su_mcs` 保持 `-1`；
- 调度后的 `actual_mcs` 使用实际 EESM SINR，而不是调度前预测 SINR。
